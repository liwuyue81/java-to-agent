⏺ LangGraph 图路由解析

  对,这段就是定义整个 Agent 的执行流程和路由规则。

  先看整体结构

  用户输入
      ↓
  supervisor ──条件路由──→ parser   ─┐
                      ──→ analyzer  ─┤── 执行完都回 supervisor
                      ──→ reporter  ─┘
                      ──→ END (结束)

  Supervisor 是"调度中心",每次执行完一个专家 Agent 后,都回到 Supervisor 决定下一步。

  逐段拆解

  1. 注册节点

  graph = StateGraph(SupervisorState)
  graph.add_node("supervisor", supervisor_node)
  graph.add_node("parser",     parser_node)

  节点 = 一个函数,接收当前 state,执行逻辑,返回新 state。

  Java 类比:像工作流引擎(Activiti/Flowable)里注册 ServiceTask:
  process.addTask("supervisor", supervisorHandler);
  process.addTask("parser", parserHandler);

  2. 固定边(无条件跳转)

  graph.add_edge("parser",   "supervisor")
  graph.add_edge("analyzer", "supervisor")
  graph.add_edge("reporter", "supervisor")

  parser 执行完无条件跳回 supervisor,analyzer/reporter 同理。

  Java 类比:sequenceFlow(顺序流),固定的箭头。

  3. 条件边(核心路由)

  graph.add_conditional_edges(
      "supervisor",                      # 从哪个节点出发
      lambda s: s["next_agent"],         # 路由函数:从 state 里取 next_agent 的值
      {
          "parser":   "parser",          # next_agent=="parser"  → 去 parser 节点
          "analyzer": "analyzer",        # next_agent=="analyzer" → 去 analyzer 节点
          "reporter": "reporter",
          "END":      END,               # next_agent=="END"     → 结束
      },
  )

  关键点:lambda s: s["next_agent"] 这个路由函数返回一个字符串,然后查 mapping
  表决定去哪个节点。

  Java 类比:像 Flowable 的排他网关(Exclusive Gateway):
  gateway.addCondition("toParser",   "${nextAgent == 'parser'}");
  gateway.addCondition("toAnalyzer", "${nextAgent == 'analyzer'}");
  gateway.addCondition("toEnd",      "${nextAgent == 'END'}");

  关键问题:next_agent 是谁写进去的?

  supervisor_node 函数执行时,会让大模型决策下一步,然后把结果写入 state:

  def supervisor_node(state):
      # 大模型分析任务 → 决定派哪个 Agent
      decision = llm.invoke(...)
      return {"next_agent": "parser"}   # 写入 state

  下一次条件边执行时,lambda s: s["next_agent"] 读到 "parser",就路由到 parser 节点。

  完整执行时序

  supervisor → next_agent="parser"  → parser
  parser     → (固定边)             → supervisor
  supervisor → next_agent="analyzer"→ analyzer
  analyzer   → (固定边)             → supervisor
  supervisor → next_agent="END"     → 结束

  这就是 Supervisor 模式的本质:一个中央调度者 + 多个专家工人,Supervisor
  每轮决定"下一个干活的是谁"。

  一句话总结

  这段代码定义了一张有向图:固定边保证专家执行完回到 Supervisor,条件边让 Supervisor 根据 state
   里的 next_agent 字段动态决定下一步去哪 —— 路由逻辑本身由大模型在 supervisor_node 里写入
  state。

  def supervisor_node(state: SupervisorState) -> dict:

      # 1. 把当前状态拼成 Prompt
      history = "\n".join(state["agent_outputs"]) or "（尚无 Agent 产出）"
      prompt = SUPERVISOR_PROMPT_TEMPLATE.format(
          user_query=state["user_query"],   # 用户原始问题
          history=history,                  # 所有 Agent 已产出的内容
          ...

      # 2. 让大模型做决策(结构化输出)
      structured_llm = llm.with_structured_output(RouteDecision)
      decision: RouteDecision = structured_llm.invoke(prompt)

      # 3. 把决策结果写入 state
      return {
          "next_agent": decision.next,    # "parser" / "analyzer" / "reporter" / "END"
          "loop_count": state["loop_count"] + 1,
      }

  决策流程图

  supervisor_node 被调用
      │
      ├── 读 state["user_query"]      用户问了什么

      │
      ├── 拼成 Prompt → 发给大模型
      │
      ├── 大模型返回 RouteDecision { next: "parser", reason: "..." }
      │

  SUPERVISOR_PROMPT_TEMPLATE 的决策规则(第 300-311 行)

  这是写给大模型的"调度规则",告诉它怎么决定:

  - agent_outputs 为空 → 通常先 parser 收集数据
  - 简单查询(数量、统计) → parser 之后直接 END
  - 含"为什么/根因"的问题 → parser 后接 analyzer
  - 用户明确要"报告/JSON" → 最后调 reporter
  - 已充分回答 → END

  大模型读完这些规则 + 当前状态,自己判断返回哪个 Agent 名字。

  RouteDecision 是什么

  大模型的输出是结构化的,大概长这样:
  class RouteDecision(BaseModel):
      next: Literal["parser", "analyzer", "reporter", "END"]
      reason: str   # 说明为什么这么路由

⏺ LangGraph 是什么 & 执行流程

  一、LangGraph 是什么

  LangGraph 是 LangChain 团队出的"Agent 编排框架",核心思想是把多个 Agent
  的协作流程定义成一张有向图。

  ┌────────┬───────────┬────────────────────────────────────┐
  │  层级  │   框架    │                职责                │
  ├────────┼───────────┼────────────────────────────────────┤
  │ 底层   │ LangChain │ 封装 LLM 调用、Tool、Prompt        │
  ├────────┼───────────┼────────────────────────────────────┤
  │ 编排层 │ LangGraph │ 把多个 Agent 组织成图,管理状态流转 │
  └────────┴───────────┴────────────────────────────────────┘

  Java 类比:
  - LangChain ≈ Spring(基础能力)
  - LangGraph ≈ Activiti/Flowable(工作流引擎)

  二、三个核心概念

  1. State —— 贯穿整个图的共享数据

  class SupervisorState(TypedDict):
      user_query:    str        # 用户问题
      agent_outputs: list[str]  # 各 Agent 产出
      next_agent:    str        # 下一步路由
      loop_count:    int        # 循环次数

  每个节点读 state、写 state,类似流水线上的"工单",每道工序在上面填内容。

  Java 类比:工作流的流程变量 ProcessVariables,或者链路上下文 TraceContext。

  2. Node —— 每个节点是一个函数

  def parser_node(state) -> dict:
      # 读 state → 干活 → 返回要更新的字段
      result = parser_agent.invoke(state["user_query"])
      return {"agent_outputs": state["agent_outputs"] + [result]}

  函数签名固定:接收 state,返回要合并进 state 的字段(不是替换整个 state)。

  3. Edge —— 节点之间的连线

  graph.add_edge("parser", "supervisor")           # 固定边:parser → supervisor
  graph.add_conditional_edges("supervisor", ...)   # 条件边:supervisor → ???

  四、graph.compile() 做了什么

  return graph.compile(checkpointer=checkpointer, interrupt_before=interrupt_before)

  编译 = 校验图的合法性 + 生成可执行对象。

  类比编译器:把你画的"流程图定义"编译成"可运行的程序"。编译后才能 .invoke() 执行。

  checkpointer 是可选的持久化机制:
  - 不传 → 无状态,跑完就结束
  - 传入 InMemorySaver → state 按 thread_id 保存,支持中断后恢复(HITL:Human In The Loop)

  │ 流程变量    │ ProcessVariables │ State (TypedDict)               │
  ├─────────────┼──────────────────┼─────────────────────────────────┤
  │ ServiceTask │ JavaDelegate     │ Node 函数                       │
  ├─────────────┼──────────────────┼─────────────────────────────────┤
  │ 排他网关    │ ExclusiveGateway │ add_conditional_edges           │
  ├─────────────┼──────────────────┼─────────────────────────────────┤
  │ 顺序流      │ SequenceFlow     │ add_edge                        │
  ├─────────────┼──────────────────┼─────────────────────────────────┤
  │ 流程实例    │ ProcessInstance  │ compiled.invoke(initial_state)  │
  ├─────────────┼──────────────────┼─────────────────────────────────┤
  │ 挂起/恢复   │ 人工任务节点     │ interrupt_before + checkpointer │
  └─────────────┴──────────────────┴─────────────────────────────────┘

  核心差异:Activiti 的路由条件是预先写死的规则表达式,LangGraph 的路由条件是LLM 实时决策 ——
  这就是"智能体"和"传统工作流"的本质区别。

  六、一句话总结

  LangGraph 是有向图工作流引擎,State 是流水线上的工单,Node 是处理工单的函数,Edge 是传递方向 ——
  和 Activiti 的架构思路完全一致,区别在于路由决策者是 LLM 而不是预设规则。

  dict.items() 返回什么

  Java 类比:

      String nodeName = entry.getKey();

  元组解包是什么

  Python 允许把元组的多个值一次性赋给多个变量:

  pair = ("supervisor", {"next_agent": "parser"})

  # 普通写法(Java 风格)
  node_name = pair[0]
  update    = pair[1]

  # Python 元组解包(一行搞定)
  node_name, update = pair

  for node_name, update in chunk.items() 就是把这两个合在一起:

  for (node_name, update) in chunk.items():
  # 括号可省略,等价上面
  for node_name, update in chunk.items():

  每次迭代,自动把元组的第0个赋给 node_name,第1个赋给 update。

  具体执行过程

  chunk = {"supervisor": {"next_agent": "parser"}}

  for node_name, update in chunk.items():
      print(node_name)  # "supervisor"
      print(update)     # {"next_agent": "parser"}

  元组解包在其他场景也很常见

  # 函数返回多个值
  def get_info():
      return "Tom", 18       # 返回元组

  name, age = get_info()     # 解包

  # enumerate(同时拿下标和值)
  for i, item in enumerate(["a", "b", "c"]):

⏺ 这段是"astream 结束后判断:是正常完成还是被 HITL 打断了"

  先理解背景                                                                                    
   
  astream 有两种结束方式:                                                                       
                  
  情况A: supervisor → parser → supervisor → END    → astream 正常结束
  情况B: supervisor → parser → supervisor → reporter(被拦!) → astream 也结束了                  
                                                                                                
  问题:两种情况 astream 循环都结束了,代码无法从循环本身区分是哪种。                             
                                                                                                
  所以需要循环结束后主动去查一下图的状态。                                                      
                  
  逐段拆解                                                                                      
                  
  第一段:主动查图的当前状态                                                                     
                  
  state_snap = compiled_graph.get_state(                                                        
      {"configurable": {"thread_id": thread_id}}
  )

  用 thread_id 去 checkpointer 里读这次执行的快照。                                             
   
  Java 类比:像 Activiti 查流程实例当前状态:                                                     
  ProcessInstance instance = runtimeService.createProcessInstanceQuery()
      .processInstanceId(processInstanceId)                                                     
      .singleResult();                                                                          
   
  第二段:判断是否中断                                                                           
                  
  if state_snap.next:   # next 不为空 = 还有待执行的节点 = 被 interrupt 拦住了
      pending_node = state_snap.next[0]   # 被拦的节点名,通常是 "reporter"                      
                                                                                                
  - state_snap.next 为空 → 图正常执行完了                                                       
  - state_snap.next 不为空 → 图在某个节点前被 interrupt_before 拦住了                           
                                                                                                
  第三段:中断 → 推 interrupt 事件给前端                                                         
                                                                                                
  yield _sse("interrupt", {                                                                     
      "thread_id":    thread_id,      # 前端拿这个去 /chat/resume                               
      "pending_node": pending_node,   # 告诉前端"reporter 等你确认"                             
      "reason":       _interrupt_reason(pending_node),  # 展示给用户看的文案                    
      "state_preview": { ... }        # 预览当前 state 给前端展示                               
  })                                                                                            
  return   # 生成器结束,等前端调 /chat/resume                                                   
                                                                                                
  然后 return 直接结束,不走下面的 done 事件。                                                   
                                                                                                
  第四段:正常结束 → 推 done 事件                                                                
                  
  answer = _summarize_answer(accumulated)
  _append_turn(session_id, query, answer)   # 存入多轮历史                                      
  yield _sse("done", {
      "final_state": accumulated,                                                               
      "session_id":  session_id,
      "turn":        len(_get_history(session_id)),                                             
  })              

  两条路径对比                                                                                  
   
  astream 循环结束                                                                              
          │       
          ├── get_state().next 不为空
          │       │                                                                             
          │       └── yield interrupt事件
          │           return (等前端 /chat/resume)                                              
          │                                                                                     
          └── get_state().next 为空
                  │                                                                             
                  └── yield done事件
                      (本轮结束,存历史)

  astream 结束不代表图执行完了,可能是被 interrupt_before
