  ---
  学习路线（按优先级）

  阶段一：巩固当前（现在）

  只有两件事需要补：

  1. 增量更新索引
  现在日志新增后需要 force=True 重建，实际项目应该只追加新内容：
  新日志 → 检查是否已索引 → 只添加新的

  2. 换 bge-m3 模型
  把 nomic-embed-text 换成 bge-m3，彻底解决中英文跨语言问题，实际体验差异很大。

  ---
  阶段二：检索质量优化（重要）

  这是 RAG 项目实际落地的核心难点：

  ┌──────────────────────────────────┬─────────────────────────────────────────┐
  │               问题               │                解决方案                 │
  ├──────────────────────────────────┼─────────────────────────────────────────┤
  │ 只用向量检索，精确词语反而找不到 │ 混合检索（向量 + 关键词同时搜，取并集） │
  ├──────────────────────────────────┼─────────────────────────────────────────┤
  │ 返回 5 条结果，有的不相关        │ 相似度阈值过滤（低于 0.7 的结果丢弃）   │
  ├──────────────────────────────────┼─────────────────────────────────────────┤
  │ 日志太长，向量化效果差           │ 分块策略（按时间窗口分块，而非按行）    │
  └──────────────────────────────────┴─────────────────────────────────────────┘

  ---
  阶段三：工程化（按需）

  如果项目要上生产才需要：

  - 持久化方案：Chroma 换 Milvus（支持并发、集群）
  - Rerank：检索后用一个小模型二次排序，提升准确率
  - 评估：用 RAGAS 框架量化检索效果（召回率、精确率）

  ---
  对你来说最值得做的下一步

  混合检索，收益最高。

  你现在纯向量检索有个典型问题：用户输入 "DBPool" 这种精确词语，语义检索反而不如关键词搜索准。混合检索同时跑两种，把结果合并，覆盖两种场景。

  LangChain 有现成的 EnsembleRetriever 直接支持，不需要从零写。
