# Prompt 工程学习路线

面向有 Java 后端经验、正在学习 AI Agent 开发的工程师。

---

## 总体原则

```
不要死记规则 → 在项目里实践 → 遇到问题再查
```

Prompt 工程没有标准答案，经验比理论重要。

---

## 第一阶段：基础入门（1-2天）

**目标**：能写出可用的 Prompt，理解基本结构。

### 学习内容

- [ ] 掌握 Prompt 五要素（角色 / 任务 / 约束 / 输入 / 输出格式）
- [ ] 理解为什么要加角色和约束
- [ ] 学会用占位符分离变量和固定文本

### 实践任务

改造本项目 `alert/monitor.py` 中的 Prompt：

```python
# 原版（缺少角色和防幻觉约束）
prompt = f"""以下是最新检测到的 ERROR 日志：
{log_text}
请用 2-3 句话简要分析：根因是什么？影响了哪些服务？建议排查方向？"""

# 你的任务：加入角色定义和约束，对比两版输出的差异
```

### 完成标准

写出的 Prompt 满足检查清单全部6条。

---

## 第二阶段：质量提升（3-5天）

**目标**：会用 Few-shot 和思维链，能处理复杂任务。

### 学习内容

- [ ] Few-shot：给示例让模型学样子
- [ ] Chain of Thought：让模型先思考再回答
- [ ] 输出格式控制：JSON / 固定模板 / 结构化输出
- [ ] 防幻觉技巧：如何让模型"不知道就说不知道"

### 实践任务

为本项目新增一个 Prompt，实现：

```
输入：多条混合级别的日志
输出（JSON格式）：
{
  "severity": "HIGH / MEDIUM / LOW",
  "root_cause": "根因",
  "affected_services": ["服务A", "服务B"],
  "suggestion": "建议"
}
```

要求：输出必须是合法 JSON，不能有多余文字。

### 完成标准

跑 20 个案例，JSON 解析成功率 100%，内容准确率 80% 以上。

---

## 第三阶段：测试体系（3-5天）

**目标**：会量化对比两版 Prompt，建立测试集。

### 学习内容

- [ ] 如何准备测试集（覆盖正常 / 边界 / 异常情况）
- [ ] 准确率测试脚本的写法
- [ ] LLM 评分方法
- [ ] 理解过拟合陷阱

### 实践任务

为本项目的日志分析 Prompt 建立测试体系：

```
1. 从 logs/app.log 中整理 30 条有代表性的日志作为测试集
2. 人工标注每条的标准答案（根因）
3. 写脚本跑准确率测试
4. 修改 Prompt，对比改前改后的准确率
```

### 完成标准

有可运行的测试脚本，能输出准确率数字。

---

## 第四阶段：结合 Agent 实战（持续）

**目标**：在 Agent 工具调用场景中设计 Prompt。

### 学习内容

- [ ] Agent 场景下 Prompt 的特殊要求（工具描述怎么写）
- [ ] System Prompt vs User Prompt 的区别
- [ ] 多轮对话中的 Prompt 管理
- [ ] RAG 场景下检索结果如何拼入 Prompt

### 实践任务

查看本项目各阶段的 Prompt 写法：

```
main.py           → 基础 Prompt
main_stage2.py    → 加入工具调用
main_stage3.py    → 结构化输出
main_stage4_a.py  → RAG + Prompt 结合
alert/monitor.py  → 告警场景 Prompt
```

逐个分析每个 Prompt 的结构，找出可以改进的地方并实际修改验证。

## 时间分配建议

```
理论学习  20%  → 看文档、看示例
动手实践  60%  → 在项目里改 Prompt、跑测试
总结反思  20%  → 记录哪些有效、哪些无效、为什么
```

---

## 关键心态

```
Prompt 没有最优解，只有更适合当前场景的解。
每次修改都要有测试数据支撑，不要凭感觉说"这版更好"。
失败的案例比成功的案例更有价值，认真分析为什么失败。
```
