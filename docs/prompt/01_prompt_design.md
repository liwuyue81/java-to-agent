# Prompt 设计指南

## 核心思想

Prompt 类比接口文档：写得清楚，LLM 就不会误解，输出结果才符合预期。

---

## 基本结构

一个完整的 Prompt 包含以下要素：

```
# 1. 角色（你是谁）
你是一个 Java 后端专家，专注于线上日志分析。

# 2. 任务（做什么）
分析以下错误日志，找出根因。

# 3. 约束（怎么做 / 不能做什么）
- 只基于提供的日志内容回答
- 不要编造日志中没有的信息
- 用中文回答

# 4. 输入数据
{log_content}

# 5. 输出格式（返回什么结构）
根因：xxx
影响服务：xxx
建议：xxx
```

---

## 五个要素说明

### 1. 角色（Role）
告诉 LLM 它扮演什么角色，影响回答的专业程度和视角。

```
# 差
帮我分析日志

# 好
你是一个有5年经验的 Java 后端工程师，擅长线上故障排查。
```

### 2. 任务（Task）
描述要做的事，越具体越好，一个 Prompt 只做一件事。

```
# 差
分析这些日志，看看有什么问题，顺便给点建议

# 好
分析以下 ERROR 日志，找出根因
```

### 3. 约束（Constraint）
限制输出范围，防止幻觉和跑题。

```
- 只基于提供内容回答，不确定就说"日志中未体现"
- 不要补充日志之外的推测
- 用中文回答
```

### 4. 输入数据（Input）
把变量用占位符表示，和固定文本分开。

```python
prompt = f"""
你是日志分析专家。
分析以下日志的根因：

{log_content}      # ← 变量部分，每次调用传入
"""
```

### 5. 输出格式（Output Format）
明确指定格式，便于后续代码解析。

```
# 要求 JSON 输出
以 JSON 格式返回：
{
  "root_cause": "根因描述",
  "affected_service": "影响的服务名",
  "suggestion": "排查建议"
}

# 要求固定行数
用 2-3 句话回答，不超过100字
```

---

## 常见问题 & 解法

| 问题现象 | 原因 | 解法 |
|---|---|---|
| 回答太啰嗦 | 没有长度约束 | 加"用3句话以内回答" |
| 输出格式不固定 | 没有格式示例 | 给出具体格式模板 |
| 编造不存在的信息 | 没有边界约束 | 加"只基于提供内容回答" |
| 忽略某个要求 | 要求位置不显眼 | 重要要求放最前或最后 |
| 中英文混乱 | 没有语言约束 | 明确指定"用中文回答" |
| 多个任务互相干扰 | 一个 Prompt 做太多事 | 拆分成多个 Prompt 串行执行 |

## 实际项目示例

### 示例1：日志根因分析（本项目）

```python
# alert/monitor.py 中的 Prompt
prompt = f"""以下是最新检测到的 ERROR 日志：

{log_text}

请用 2-3 句话简要分析：根因是什么？影响了哪些服务？建议排查方向？"""
```

分析：
- 有输入数据（log_text）
- 有长度约束（2-3句话）
- 有明确的输出要求（三个问题）
- 缺少角色定义（可以加上）

### 示例2：加强版（加入角色和约束）

```python
prompt = f"""你是一个有丰富经验的后端运维工程师。

以下是系统检测到的 ERROR 日志：

{log_text}

请基于以上日志（不要补充日志之外的推测），用 2-3 句话分析：
1. 根因是什么？
2. 影响了哪些服务？
3. 建议排查方向？

用中文回答。"""
```

## 进阶技巧

### Few-shot：给示例
当任务复杂时，给 1-3 个输入输出的示例，效果大幅提升。

```
任务：提取日志中的服务名

示例1：
输入：2024-01-01 ERROR UserService - 连接超时
输出：UserService

示例2：
输入：2024-01-01 WARN OrderService - 库存不足
输出：OrderService

现在处理：
输入：{log_line}
输出：
```

### 思维链（Chain of Thought）
复杂推理任务，让模型先说思路再给答案：

```
请一步步分析，先列出观察到的异常点，再推断根因，最后给出建议。
```

### 输出 JSON 防止解析失败

```
只输出 JSON，不要有任何其他文字，不要有 markdown 代码块。
```
