# 第五阶段技术方案

## 一、异常自动告警

### 核心问题

前四个阶段都是「你问 Agent 才回答」，告警是反过来的——**Agent 主动监控，发现问题主动通知你**。

### 业界标准流程

```
定时触发
    ↓
读取最新日志（增量，不是全量）
    ↓
规则检测：最近 5 分钟 ERROR 数 > 阈值？
    ↓ 是
LLM 分析根因，生成告警摘要
    ↓
推送通知（钉钉 / 邮件 / Slack / 企业微信）
    ↓
记录「已告警」，避免重复推送
```

### 关键设计点

**1. 增量读取**

不能每次都读全量日志，要记录上次读到哪一行（offset），下次从那里继续。类比 MySQL binlog 的位点消费。

**2. 告警规则**

两类规则结合使用：

| 规则类型 | 实现方式 | 特点 |
|---|---|---|
| 硬规则 | 代码写死：5 分钟内 ERROR > 5 条触发 | 不依赖模型，快且准 |
| 软规则 | LLM 判断：把最新日志给模型，问「有没有异常趋势」 | 能发现硬规则覆盖不到的场景 |

**3. 告警去重**

同一个问题不能每分钟都推一条。用一个简单的状态文件（或内存 dict）记录「DBPool 连接池耗尽」已在某时间告警过，冷却期内不重复推送。

**4. 推送渠道**

最简单的是钉钉 / 企业微信 Webhook，一行 HTTP 请求就能发消息，不需要注册账号，本地测试也方便。

## 二、LangGraph

### 它解决什么问题

前四个阶段用的 `create_react_agent` 是一个**线性循环**：思考 → 调 Tool → 观察 → 思考... 直到结束。

这个结构有局限：
- 无法并行调多个 Tool
- 无法做「如果 A 失败就走 B 分支」的条件跳转
- 无法让多个 Agent 协作分工

LangGraph 把 Agent 的执行过程建模成一张**有向图**，每个节点是一个步骤，边是跳转条件。

```
                 ┌─────────────┐
                 │  收集日志数据 │
                 └──────┬──────┘
                        │
           ┌────────────┼────────────┐
           ↓            ↓            ↓
    ┌─────────────┐ ┌─────────┐ ┌──────────┐
    │ 统计 ERROR  │ │分析 WARN│ │搜索关键词│  ← 并行执行
    └──────┬──────┘ └────┬────┘ └────┬─────┘
           └────────────┬┘           │
                        ↓            │
                 ┌─────────────┐     │
                 │  汇总分析结果 │←───┘
                 └──────┬──────┘
                        │
              ┌─────────┴──────────┐
              ↓                    ↓
       有严重异常？              无异常
              ↓                    ↓
       ┌────────────┐        ┌──────────┐
       │ 触发告警流程 │        │ 输出报告 │
       └────────────┘        └──────────┘
```

### 对比 ReAct Agent

| | ReAct Agent（前四阶段）| LangGraph |
|---|---|---|
| 执行方式 | 线性循环 | 有向图，支持分支、并行 |
| 适合场景 | 单一任务，步骤不确定 | 复杂流程，步骤明确可设计 |
| 调试难度 | 难（模型自由发挥）| 易（流程可视化，每个节点可单独测）|
| 上手难度 | 低 | 中 |

### 在日志 Agent 里怎么用

把「告警系统」用 LangGraph 实现是最自然的结合点：

```
节点1：读取增量日志
节点2：并行分析（ERROR统计 + WARN统计 + 根因分析）
节点3：判断节点（有无异常）→ 分支
节点4a：告警节点（生成摘要 + 推送）
节点4b：正常报告节点
```

## 建议的实施方案

**先做告警，再做 LangGraph：**

1. 告警系统用现有 ReAct Agent 先跑通（定时触发 + 增量读取 + 推送）
2. 理解 LangGraph 基本概念（节点、边、状态）
3. 把告警系统的流程用 LangGraph 重写，对比两种写法的差异

这样 LangGraph 不是凭空学，而是有具体场景对照，理解会快很多。
